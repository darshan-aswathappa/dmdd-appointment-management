Database Users & Roles

Three Oracle users are provisioned for the HMS schema, each designed for a different operational concern. The dropdown in the top-right switches the UI's active connection between them so the same UI demonstrates what each role can and cannot do. Permission failures raised by Oracle are surfaced in the flash banner as “Permission denied as <user>”.

You are currently connected as {{ current_user.user }} — role {{ current_user.role }}. Use the dropdown in the top nav to switch; your choice is stored in the browser session and persists until you switch again.

Why three users?

The HMS schema follows the Oracle convention of separating object ownership from application access. Mixing the two is a common security footgun: if the application connects as the owner, any SQL-injection or privilege-escalation bug instantly has DROP TABLE power. Splitting the credentials means the blast radius of an application-layer compromise is bounded by whatever the specific role is allowed to do.

The three users

HMS (schema owner)

Role: OWNER · Credentials: HMS / HospitalMS@Pass123

Owns every object in the HMS schema — tables, indexes, views, procedures, triggers, and sequences. Has system privileges (CREATE TABLE, CREATE PROCEDURE, etc.) and unlimited tablespace.

When to use it

  • Running migrations and DDL scripts from DDL/.
  • Seeding data from DML/.
  • Creating and altering procedures, triggers, and views.
  • Local development and schema maintenance.

When not to use it

  • Serving user traffic from an application. The principle of least privilege says the app should never hold DDL rights.

admin_user (HospitalAdmin role)

Role: HospitalAdmin · Credentials: admin_user / AdminPass123

The application-tier admin. Owns nothing but holds the HospitalAdmin role, which grants SELECT, INSERT, UPDATE, DELETE on every HMS table, plus SELECT on all HMS views, plus EXECUTE on the appointment procedures.

When to use it

  • Running the appointment flows (book, cancel, reschedule, complete).
  • CRUD work from the Tables section.
  • Any backend service that needs to mutate data.

What it cannot do

  • Create, alter, or drop any object (no DDL privileges).
  • Grant privileges to other users.

Unqualified table references are resolved via ALTER SESSION SET CURRENT_SCHEMA = HMS on connect, so the same SQL works for all three users.

operator_user (HospitalOperator role)

Role: HospitalOperator · Credentials: operator_user / OperatorPass456

The least-privilege operator. Holds the HospitalOperator role, which grants SELECT on all HMS tables and views and execute permissions to — no INSERT, no UPDATE, no DELETE.

When to use it

  • Reporting, dashboards, and analytics consumers.
  • Support staff who need visibility but must not be able to mutate data.
  • Least-privilege integrations (exports, BI tools).
  • Running appointment flows (book / cancel / reschedule / complete) without granting raw table-write privileges.

Expected permission errors

  • Any CRUD Add / Edit / Delete action → ORA-01031: insufficient privileges.
  • Views, Reports, and browse pages continue to work.

Permission matrix

Operation HMS admin_user operator_user
DDL (create/alter/drop tables, procedures, triggers) yes no no
SELECT on tables and views yes yes yes
INSERT / UPDATE / DELETE on tables yes yes no
EXECUTE on appointment procedures yes yes yes
Run saved reports (SELECT-based) yes yes yes

Demo walkthrough

  1. Switch to operator_user via the top-right dropdown.
  2. Go to Tables → any table → Add. Submit the form → flash shows “Permission denied as operator_user - ORA-01031: insufficient privileges” (direct DML is blocked).
  3. Run an appointment flow (Flows → Book) → succeeds. This is the teaching moment: the procedure path is allowed even though raw table writes are not — the role has execute access, not DML access.
  4. Open a Report → works normally (reports are SELECT-only).
  5. Switch back to admin_user or HMS → rerun the direct table write and it succeeds (raw DML is restored).

Source of truth for the role definitions: DDL/01_create_database.sql (users) and DDL/06_create_roles_security.sql (roles and grants).
