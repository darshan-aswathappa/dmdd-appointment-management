{% if tab == 'procedures' %}

Stored procedures encapsulate the core business flows — booking, cancelling, rescheduling, and completing appointments — and enforce the validation rules that cannot be expressed as simple constraints.

sp_BookAppointment PROCEDURE

File: Procedures/sp_BookAppointment.sql

Books a new appointment for a patient at a given DoctorSchedule slot. Validates the patient, slot availability, doctor vacation status, and duplicate bookings before inserting the row, then returns the new appointment_id.

p_patient_id
IN NUMBER — patient requesting the appointment
p_schedule_id
IN NUMBER — doctor schedule slot to book
p_visit_type
IN VARCHAR2 — 'In-Person' or 'Teleconsultation'
p_reason
IN VARCHAR2 — free-text reason
p_appointment_id
OUT NUMBER — new appointment ID on success

Raises:

  • ORA-20001 — patient does not exist
  • ORA-20002 — schedule slot missing or not available
  • ORA-20003 — doctor is on approved vacation that day
  • ORA-20004 — patient already has a Scheduled appointment in that slot

sp_CancelAppointment PROCEDURE

File: Procedures/sp_CancelAppointment.sql

Cancels a Scheduled appointment if it is at least 24 hours away. Sets status = 'Cancelled', stamps cancelled_at, and stores the cancellation reason in notes. The history row is written automatically by trg_AppointmentHistory.

p_appointment_id
IN NUMBER — appointment to cancel
p_cancellation_reason
IN VARCHAR2 — reason (stored in notes)

Raises:

  • ORA-20001 — appointment missing or not in Scheduled status
  • ORA-20002 — attempting to cancel within 24 hours of the slot (BR2)

sp_RescheduleAppointment PROCEDURE

File: Procedures/sp_RescheduleAppointment.sql

Moves an existing Scheduled appointment to a new slot. Validates the new slot, checks the doctor's vacation calendar, prevents duplicate bookings, and writes an explicit row into AppointmentHistory carrying the caller-supplied change_reason (which a status-only trigger cannot see).

p_appointment_id
IN NUMBER — appointment to reschedule
p_new_schedule_id
IN NUMBER — target slot
p_change_reason
IN VARCHAR2 — reason logged into history

Raises:

  • ORA-20001 — appointment missing or not Scheduled
  • ORA-20002 — new slot missing or not available
  • ORA-20003 — doctor on approved vacation on the new date
  • ORA-20004 — patient already booked in the target slot

sp_CompleteAppointment PROCEDURE

File: Procedures/sp_CompleteAppointment.sql

Marks a Scheduled appointment as Completed and writes embedded billing fields (bill_amount, billing_status, billing_date, due_date) onto the same row. The trg_AppointmentInsDiscount trigger then fires and automatically populates insurance_coverage_amt.

p_appointment_id
IN NUMBER — appointment to complete
p_notes
IN VARCHAR2 (optional) — clinical notes
p_bill_amount
IN NUMBER — total bill for the visit

Raises:

  • ORA-20001 — appointment missing or not Scheduled

Billing lives on Appointment and Admission; this procedure does not insert into a separate billing table.

sp_GetDoctorAvailability PROCEDURE

File: Procedures/sp_GetDoctorAvailability.sql

Opens a ref cursor of available schedule slots for a doctor within a date range. Excludes slots with is_available = 0, dates overlapping an approved vacation, and slots that already carry a Scheduled appointment.

p_employee_id
IN NUMBER — employee/doctor to query
p_start_date
IN DATE — inclusive range start
p_end_date
IN DATE — inclusive range end
p_cursor
OUT SYS_REFCURSOR — (schedule_id, schedule_date, slot_time, slot_duration_mins, current_bookings)

Functions are pure lookups used by procedures and triggers. They return a value and do not modify data.

fn_IsDoctorOnVacation FUNCTION

File: Procedures/fn_IsDoctorOnVacation.sql

Returns whether an employee of type Doctor has an approved vacation that covers the supplied date. Used by sp_BookAppointment and sp_RescheduleAppointment to prevent booking during time off.

p_employee_id
IN NUMBER — doctor to check
p_check_date
IN DATE — target date
RETURNS
NUMBER(1) — 1 if on approved vacation, 0 otherwise

A vacation is considered active when status = 'Approved' and p_check_date BETWEEN start_date AND end_date.

{% endif %} {% if tab == 'triggers' %}

Triggers enforce invariants the application cannot be trusted to maintain: audit history, insurance calculations, and hard business rules like “no two active admissions on the same bed.”

trg_AppointmentHistory TRIGGER

File: Triggers/trg_AppointmentHistory.sql · AFTER UPDATE ON Appointment (row-level)

Captures every status transition on Appointment into AppointmentHistory. It records the previous status, new status, previous schedule_id, and a snapshot of the old slot_time taken from DoctorSchedule.

When it fires: only when :OLD.status <> :NEW.status. Pure schedule_id changes are intentionally ignored here because sp_RescheduleAppointment writes its own history row with the caller-provided change_reason, and firing on both would double-log the audit entry.

trg_AppointmentInsDiscount TRIGGER

File: Triggers/trg_AppointmentInsDiscount.sql · BEFORE UPDATE OF bill_amount ON Appointment (row-level, WHEN NEW.bill_amount IS NOT NULL)

Automatically computes insurance_coverage_amt whenever bill_amount is first set or subsequently changed. Looks up the patient's active insurance policy and applies coverage_percentage, capped at max_coverage_amount.

Inactive or expired policy (is_active = 0 or valid_until < TRUNC(SYSDATE)) → coverage is set to 0. No policy linked → coverage is 0.

Design: fires only on UPDATE so seed rows that ship pre-computed coverage are not overwritten. Uses a BEFORE trigger with direct :NEW assignment to avoid the mutating-table problem that an AFTER UPDATE + self-UPDATE would cause.

trg_AdmissionInsDiscount TRIGGER

File: Triggers/trg_AdmissionInsDiscount.sql · BEFORE UPDATE OF bill_amount ON Admission (row-level)

Mirror of trg_AppointmentInsDiscount for the admission side of the billing fold-in. Applies the same insurance calculation whenever an admission's bill_amount is set or changed.

trg_PreventOccupiedBed TRIGGER

File: Triggers/trg_PreventOccupiedBed.sql · BEFORE INSERT ON Admission (row-level)

Hard-blocks admitting a patient to a bed that already carries an Active admission. Counts existing active admissions on :NEW.bed_id and raises ORA-20002 'Bed is currently occupied...' to reject the insert.

Oracle does not support INSTEAD OF triggers on tables, so this is implemented as a BEFORE INSERT guard that aborts the transaction rather than rewriting the row.

{% endif %} {% if tab == 'reports' %}

Reports are saved SQL queries used for operational and financial reporting. They run read-only against the schema and, where noted, rely on the v_billing view to unify appointment and admission billing.

rpt_DailyAppointments REPORT

File: Reports/rpt_DailyAppointments.sql

Lists every appointment scheduled for a given date with patient, doctor (via Employee filtered to employee_type = 'Doctor'), department, slot time, visit type, status, and reason.

Parameter: report_date (DATE) — default in the file is 2026-04-14.

Columns: AppointmentID, PatientName, DoctorName, DepartmentName, ScheduleDate, SlotTime, SlotDurationMins, VisitType, Status, Reason.

rpt_DoctorSchedule REPORT

File: Reports/rpt_DoctorSchedule.sql

Per-doctor schedule view for a date range. Shows each slot with its current booking count and whether that day collides with an approved vacation.

Parameters: start_date, end_date (DATE).

Columns: DoctorName, Specialization, Department, ScheduleDate, SlotTime, DurationMins, CurrentBookings, OnVacation (Y/N).

The CurrentBookings column reflects the number of Scheduled appointments on each slot; there is no hard daily cap per doctor.

rpt_BedOccupancy REPORT

File: Reports/rpt_BedOccupancy.sql

Two queries:

  • Detailed Bed Occupancy — every bed on every active room with its department, status (Maintenance / Occupied / Available), and current patient when occupied.
  • Occupancy Summary — hospital-wide rollup: total beds, occupied, available, in maintenance, and OccupancyPercent.

A bed is flagged Occupied when an Admission with status = 'Active' points to it.

rpt_Revenue REPORT

File: Reports/rpt_Revenue.sql

Three queries over v_billing (which unifies appointment and admission billing after the Billing-table fold-in):

  • Overall Revenue Summary — TotalBilled, TotalInsuranceCoverage, NetPatientLiability, TotalCollected (from Payment with status = 'Completed'), and OutstandingBalance.
  • Revenue By Source — same metrics grouped by Appointment vs Admission.
  • Revenue By Billing Status — breakdown across Pending, Partially Paid, Overdue, Paid.

Parameters: start_date, end_date (DATE).

Payments are aggregated per source entity via (appointment_id, admission_id) keys since the old bill_id FK was retired with the Billing table.

rpt_CancellationStats REPORT

File: Reports/rpt_CancellationStats.sql

Per-doctor cancellation analysis for a date range. Uses two CTEs — one for appointment counts and cancellation averages, one to rank each doctor's most common cancellation reason from AppointmentHistory.change_reason — then joins them into the final result.

Parameters: start_date, end_date (DATE).

Columns: DoctorName, Department, TotalAppointments, CancelledCount, CancellationRatePct, MostCommonReason, AvgDaysBeforeAppt.

Doctors are modeled as Employee rows filtered by employee_type = 'Doctor'.
