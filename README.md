## Project Overview

This project implements the database design from Part A as a fully functional Oracle database for a Hospital Management System. The primary focus is the **Appointment Management** module, covering the complete appointment lifecycle: booking, cancellation, rescheduling, and status tracking. Cross-cutting concerns including bed management, billing with insurance integration, and role-based security are also implemented.

### Schema revamp (v2)

The original design has been consolidated:

1. **Patient + Guardian** — Guardian info is embedded in `Patient` (guardian_name, guardian_phone, guardian_relationship). No separate Guardian table.
2. **Max appointments per day (BR4) retired** — `DoctorSchedule.max_appointments` and the related check, procedure validations, function (`fn_GetDoctorDailyCount`), and test case (TC03) have been removed.
3. **Doctor + Staff → Employee** — A single `Employee` table with an `employee_type` discriminator replaces the two tables. All FKs retargeted to `Employee.employee_id`.
4. **No AdmissionHistory** — Never existed; only `AppointmentHistory` is kept.
5. **Billing folded into Appointment + Admission** — The old `Billing` table has been replaced with embedded billing columns (`bill_amount`, `insurance_coverage_amt`, `billing_status`, `billing_date`, `due_date`) on both `Appointment` and `Admission`. `Payment` now references the source entity directly (appointment_id or admission_id, XOR).

## Module Scope

| Function | Description |
|----------|-------------|
| Book Appointments | Schedule a patient with a doctor on an available slot |
| Cancel Appointments | Cancel with 24-hour advance rule enforcement |
| Reschedule Appointments | Move to a new slot while preserving full history |
| Status Tracking | Scheduled, Completed, Cancelled, No-Show |
| Billing on Completion | `sp_CompleteAppointment` writes billing fields directly on the Appointment row; insurance coverage is computed by `trg_AppointmentInsDiscount` |

## How to Run

See [MASTER_EXECUTION_GUIDE.sql](MASTER_EXECUTION_GUIDE.sql) for the step-by-step playbook. In summary:

1. Run DDL scripts 01-02 as **SYS/SYSTEM** (DBA), then 03-05 as **HMS** user; run 06 as **DBA**.
2. Alter identity columns to `GENERATED BY DEFAULT`, then run all DML scripts (01, 02, 04, 05, 06, 07, 08, 09) as **HMS** in order.
3. Create functions first, then stored procedures, then triggers as **HMS**.
4. Execute reports as **HMS** or **operator_user**.
5. Run test cases as **operator_user** to verify permissions and business rules.
6. Run `SET SERVEROUTPUT ON;` before test cases to see DBMS_OUTPUT messages.

## Business Rules

| Rule | Description | Enforcement |
|------|-------------|-------------|
| BR1 | No duplicate booking (same patient + slot) | Stored Procedure + UNIQUE constraint |
| BR2 | Cancellation only 24 hours before appointment | Stored Procedure date arithmetic check |
| BR3 | Rescheduling preserves full history | Procedure INSERT + Trigger |
| ~~BR4~~ | ~~Max 5 appointments per doctor per day~~ | **RETIRED in revamp** |
| BR5 | No booking during doctor vacation | Stored Procedure + `fn_IsDoctorOnVacation` |
| BR6 | Doctor schedule cannot overlap | UNIQUE constraint |
